The Problem It Solves

Publishing one SEO article manually takes 4 to 8 hours. The steps are: pick a keyword, research top-ranking content, write 2,000+ words, apply on-page SEO, find and embed images, format headings, write a meta title and description, and finally publish. Most people outsource this work at $50 to $200 per article. AI writing tools reduced the writing time, but they did not remove the other steps. You still had to handle SEO, find media, format everything, and publish it yourself.

Nany Article removes all of those steps. You type a keyword and click Generate. In under 60 seconds you get a complete, published article.

How the System Works

The full pipeline runs in this order:

  1. Keyword input: The user enters a keyword or topic into the Next.js frontend and clicks Generate.

  2. Research phase: The backend calls Google Custom Search API to fetch the top 5 to 10 ranking URLs for that keyword. The content from those pages is extracted and summarized to understand what topics, headings, and depth competitors are using.

  3. Outline generation: GPT-4o receives the research summary and generates a structured article outline with H2 and H3 headings, word count targets per section, and keyword placement instructions.

  4. Parallel section writing: Each section of the outline is sent to GPT-4o as a separate API call. These calls run in parallel using Promise.all(), which means a 10-section article takes the same time as writing one section rather than ten times longer. Each section prompt includes the keyword, the target word count, the section heading, and context from the previous section to maintain coherence.

  5. SEO layer: After all sections are assembled, a separate GPT-4o call generates the meta title, meta description, and a focus keyword list. Keyword density is checked programmatically. Image alt text is generated per image. Headings are validated for correct hierarchy.

  6. Media fetching: The Pexels API is called with the article topic to fetch 3 to 5 royalty-free images. The YouTube Data API is called to find 1 to 2 relevant videos. Images are inserted at section breaks. Videos are embedded at the most relevant heading.

  7. Publishing: The assembled HTML is sent to the WordPress REST API. The system authenticates with an application password, sets the post status to published, assigns a featured image from the Pexels results, and returns the live URL.

  8. Live progress: Throughout every step, Socket.io emits events to the frontend. The user sees each step complete in real time: Research done, Outline ready, Writing section 3 of 8, Adding images, Published.

Redis caches research results by keyword hash. If the same keyword is run again within 24 hours, the research phase is skipped. This reduces API costs and brings total time down from 60 seconds to under 20 seconds on cached runs.

The Biggest Engineering Challenge

The first version generated the entire article in a single GPT-4o call. This created two problems. First, GPT-4o has a maximum output token limit, so articles over 2,000 words were cut short. Second, the entire API call had to finish before the user saw anything, which meant a blank screen for 30 to 90 seconds with no feedback.

Both problems were solved with the same architectural change: split the article into sections and generate each section as a separate parallel API call.

Each section call targets 200 to 400 words, which is well within GPT-4o's reliable output range. Running 8 to 10 section calls in parallel with Promise.all() means the total generation time is determined by the slowest single call, not the sum of all calls. The result is faster generation, no token limit issues, and a system that can produce articles of any length by simply adding more sections to the outline.

Socket.io was added as the second part of the solution. Each section call emits a progress event when it finishes. The user sees live feedback as each section completes, so the wait feels short even when the total time is 45 to 60 seconds.
