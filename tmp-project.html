Nany Article is a fully automated blog writing system. Enter one keyword, click Generate, and within 60 seconds a 3,000-word SEO-optimized article, complete with images, video embeds, and meta tags, is published live on WordPress. No manual editing. No formatting. No publishing step.

Problem It Solves

Writing one quality blog post manually takes 3 to 6 hours: keyword research, drafting, SEO tuning, image sourcing, formatting, and publishing. Outsourcing to a freelance writer costs $50 to $200 per article. AI writing tools reduced the writing time, but they did not remove the other steps. You still had to handle SEO, find media, format everything, and publish it yourself.

Nany Article removes all of those steps. You type a keyword and click Generate. In under 60 seconds you get a complete, published article at a cost of up to 200 times less than hiring a writer.

How It Works

The system runs a sequential pipeline from keyword input to published post. Independent API calls run in parallel to keep total time under 60 seconds. Socket.io emits a progress event at each step so the user sees live feedback throughout.

  1. Keyword input: The user enters a target keyword in the Next.js interface and clicks Generate.

  2. Research: The backend calls Google Custom Search API to fetch the top 5 to 10 ranking URLs for that keyword. The content from those pages is analyzed to identify subtopics, heading patterns, and content depth used by competitors.

  3. Outline generation: GPT-4o receives the research summary and produces a structured outline: H2 and H3 headings, target word count per section, and keyword placement instructions.

  4. Parallel section writing: Each section of the outline is sent to GPT-4o as a separate API call. All section calls run in parallel using Promise.all(). Each prompt includes the keyword, target word count, section heading, and a summary of the previous section to maintain coherence. A 10-section article takes the same time as writing one section, not ten times longer.

  5. SEO processing: After all sections are assembled, a separate call generates the meta title and meta description. Keyword density is checked programmatically. Headings are validated for correct H1/H2/H3 hierarchy. Image alt text is generated per image.

  6. Media fetching: Pexels API is called with the article topic to fetch 3 to 5 royalty-free images. YouTube Data API finds 1 to 2 relevant videos. Images are inserted at section breaks. Videos are embedded at the most relevant heading position.

  7. WordPress publishing: The assembled HTML is sent to the WordPress REST API using application password authentication. The system sets post status to published, assigns a featured image, adds categories and tags, and returns the live post URL.

Redis caches research results by keyword hash for 24 hours. On a repeated keyword, the research step is skipped entirely. This brings total time down from 60 seconds to under 20 seconds and eliminates the Google Custom Search API cost for that run.

Biggest Challenge

The first version generated the entire article in a single GPT-4o API call. This caused two problems. First, GPT-4o has a maximum output token limit, so articles above 2,000 words were cut short mid-sentence. Second, the user saw a blank screen for 30 to 90 seconds with no indication of progress. Early end-to-end times reached 3 to 5 minutes.

Google Search (5s) → GPT-4o outline (10s) → GPT-4o full article in one call (45s) → Pexels (3s) → YouTube (3s) → WordPress upload (5s) = over 3 minutes total, article cut short above 2,000 words.

How I Solved It

The pipeline was redesigned around two changes: parallel section generation and real-time progress events.

  • Section-based generation: Instead of one large GPT-4o call, the article is split into sections at the outline stage. Each section is generated in a separate API call targeting 200 to 400 words, which is well within GPT-4o's reliable output range. All section calls run in parallel with Promise.all(), so a 10-section article finishes in the time it takes to generate one section.

  • Parallel media fetching: While GPT-4o writes the article sections, Pexels and YouTube API calls run in parallel using topic data from the research phase. These calls no longer wait for article generation to finish.

  • Redis caching: All external API responses are cached by keyword hash. On a repeated keyword, the research and media steps return from cache instantly, removing those calls from the critical path entirely.

  • Socket.io progress events: Each completed section emits a WebSocket event to the browser. The user sees "Writing section 3 of 8" update in real time rather than waiting with no feedback.

Result: total pipeline time dropped from over 3 minutes to under 60 seconds for a 3,000-word article with full SEO, images, and video, with no change to any individual API or model. The token limit issue was eliminated entirely because no single API call exceeds 400 words of output.
