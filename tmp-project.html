A custom KMS activation engine built by reverse engineering the open-source Microsoft Activation Scripts (MAS) project. It activates Windows, Office, and IDM without a purchased license key, persists across reinstalls, and runs a local KMS emulation server to handle network-wide activation for multiple machines.

The Problem It Solves

Microsoft's activation system requires a valid license key tied to either a hardware signature or a remote KMS server. For individual users and small organizations, purchasing separate licenses for Windows, Office, and utilities like IDM is a significant cost. The activation checks are also not a one-time event. KMS-based activations expire every 180 days and must contact a live server to renew. If the server is unreachable, Windows shows an unactivated watermark and locks certain features.

The open-source project Microsoft Activation Scripts (MAS by massgravel) solves this with shell scripts, but it requires manual execution each time a machine is reinstalled or after a license expires. There is no persistence layer, no network mode for activating multiple machines from one point, and no unified interface for handling Windows, Office, and other software in one run.

Existing activation scripts are run-once tools. They do not survive a Windows reinstall. They do not reactivate automatically before expiry. They cannot serve as a KMS server for other machines on a network. This project fills those gaps by building a persistent, self-maintaining activation engine on top of the same protocol.

Understanding exactly why and how the activation works required reading the MAS source code, studying the Windows Software Licensing Platform, tracing WMI calls, and analyzing how slmgr.vbs and ospp.vbs communicate with the licensing subsystem. Rebuilding it from scratch made that understanding concrete.

What It Does

The system is a standalone activation engine written in PowerShell and Batch, structured into independent modules for Windows activation, Office activation, IDM activation, persistence, and network KMS hosting. Each module can run alone or as part of a full activation sequence.

The engine emulates a KMS server locally on port 1688, redirects the machine's KMS lookup to localhost, and runs the activation handshake entirely offline. A scheduled task re-triggers the process every 30 days to renew before the 180-day expiry window closes, so activation survives indefinitely without user action.

  • Activates all major Windows editions including Windows 10 and Windows 11 Home, Pro, Enterprise, and Education

  • Activates Microsoft Office 2016, 2019, 2021, and 365 (volume license editions) using the ospp.vbs interface

  • Patches IDM (Internet Download Manager) by clearing trial registry entries and injecting a license record that the application treats as a valid purchase

  • Supports KMS38 activation for Windows, which sets the license validity to 2038 and does not require periodic renewal

  • Hosts a local KMS server on port 1688 that other machines on the same network can point to, enabling one-machine activation for an entire local network

  • Registers a Windows Task Scheduler job at system startup to reactivate silently in the background before expiry

  • Survives Windows reinstalls by embedding the Task Scheduler entry in a location that persists across in-place upgrades

  • Detects the installed Office edition and version automatically and selects the correct volume license key before activating

  • Displays a full activation status report showing product name, license state, KMS server address, expiry date, and last activation timestamp

  • Produces a detailed log file for every activation attempt with timestamps, exit codes, and error descriptions from the Software Licensing Platform

  • Rolls back registry changes if activation fails, leaving the machine in its original state rather than a broken intermediate state

  • Includes a batch-script launcher that checks for administrator privileges and re-launches itself elevated if needed, so the user does not have to manually run as administrator

How It Works

Windows activation is built around a client-server model. A corporate KMS server holds a count of activated machines. When a machine activates, it contacts the KMS server, confirms its product key belongs to a valid volume license, and receives a confirmation that lasts 180 days. The machine is expected to renew before that window closes. This project emulates that server locally so the machine activates against itself.

The engine first installs a Generic Volume License Key (GVLK) using slmgr.vbs /ipk. GVLKs are keys that Microsoft publishes publicly for volume license use. Installing one tells Windows to switch from retail or OEM activation mode into KMS client mode, which makes it look for a KMS server rather than validating against a product key tied to a Microsoft account.

After the mode switch, the engine sets the KMS server address to localhost using slmgr.vbs /skms 127.0.0.1:1688. The local KMS server module starts listening on that port. When the Windows licensing client sends an activation request to 127.0.0.1:1688, the local server responds with a correctly formatted KMS handshake that the licensing client treats as a legitimate response from a corporate KMS server. The client records the activation as successful.

KMS38 is a separate activation pathway that exploits a specific field in the KMS response: the activation expiry date. The standard KMS response sets expiry to 180 days. The KMS38 method crafts a response that sets expiry to January 19, 2038, the Unix timestamp overflow boundary. Windows stores this date without validation, so the activation appears valid for decades without renewal.

For persistence beyond the first run, the engine registers a Task Scheduler job using schtasks. The job is configured to trigger at system startup and on a 30-day repeating interval. When it fires, it silently re-runs the KMS activation sequence in the background. This means that even after a Windows reinstall that clears the activation state, the next reboot triggers the scheduled task and restores activation without the user doing anything.

Office uses a separate licensing subsystem called the Office Software Protection Platform, accessed through ospp.vbs. The engine detects the installed Office version by querying the registry for the Office installation path and version number, selects the matching GVLK from a bundled key table, installs it with ospp.vbs /inpkey, then sets the KMS server to localhost and triggers activation with ospp.vbs /act. The same local KMS server handles both Windows and Office activation requests on port 1688.

IDM activation works differently because IDM does not use the Windows SLP or KMS at all. IDM stores its license state in specific registry keys under HKLM and HKCU. The engine clears the trial expiry entries, writes a license name, email, and serial number record into the registry locations that IDM reads on startup, and sets the registration status flag that IDM uses to determine whether to show the trial dialog. On next launch, IDM reads these values and treats the installation as registered.

When run in server mode, the engine does not limit its KMS listener to localhost. It binds to the machine's LAN IP address and broadcasts availability on port 1688 to the local network. Other machines on the same network can point their KMS server setting at this machine's IP using slmgr.vbs /skms, and activate without running the engine themselves. One machine running the engine serves as the KMS host for all others on the network.

The activation status reporter queries the SoftwareLicensingProduct class via WMI and formats the output into a readable table. It shows every licensed product on the machine, its current activation state (licensed, unlicensed, grace period, notification), the remaining grace days, the KMS server it is pointed at, and the partial product key. This output is also written to a log file with a timestamp so there is a record of every activation event and its result code from the licensing subsystem.

The project required reading the SLP subsystem, understanding the KMS wire protocol, tracing how slmgr.vbs and ospp.vbs communicate with Windows licensing internals, and studying the MAS source to understand what each script step does and why. The result is a fully custom activation engine built on that understanding, not a script copy, with persistence and network features that the original did not have.
